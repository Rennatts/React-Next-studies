# Server-side in Next.js (App Router)

In the **App Router**, **React Server Components (RSC)** are the default: most components are **server components** unless marked with **`"use client"`**. That single rule drives **where hooks run**, **what can be async**, **how `fetch` is cached**, and **how secrets stay off the wire**.

## What runs on the server?

- **Every module in the Server Component tree** — data access, ORM calls, reading **`cookies()`** / **`headers()`**, filesystem reads (where allowed), and heavy computation that never needs to ship to the browser.
- **Server Actions** — functions marked with **`"use server"`** (inline on an action or in a separate module) invoked from forms or from Client Components; the RPC boundary is generated by Next.
- **Route Handlers** — `app/api/.../route.ts` (or `route.js`) are always server-side HTTP handlers, not React trees.

**Client Components** (`"use client"`) still render **on the server once** for HTML, then hydrate on the client; interactive behavior and browser-only APIs live there.

## Async Server Components

Server Components may be **`async`** and **`await`** data directly in the component body:

```tsx
export default async function Page() {
  const data = await getData();
  return <main>{/* ... */}</main>;
}
```

That replaces many `getServerSideProps`-style patterns from the Pages Router. **Do not** call `async` server logic from Client Components without going through **Server Actions**, **Route Handlers**, or **props passed down** from a parent Server Component.

A **runnable** version (async component + `fetch` + `next.revalidate`) lives in [`ServerSideRscFetchDemo.tsx`](./ServerSideRscFetchDemo.tsx) and is rendered on the [study page](/studies/next-server/server-side).

## `fetch` and caching (App Router)

- **Default** — `fetch` in Server Components is **cached** (`force-cache` semantics) unless you opt out.
- **`cache: 'no-store'`** — always dynamic, no static reuse.
- **`next: { revalidate: N }`** — **ISR-style** revalidation in seconds.

Treat these as **part of your server data contract**, not “just fetch”: they affect build output, CDN behavior, and staleness.

## Dynamic rendering signals

Using certain APIs forces **dynamic** rendering for the route (or segment), for example:

- **`cookies()`**, **`headers()`**, **`searchParams`** (on pages), **`connection()`**, **`draftMode()`**
- **`unstable_noStore()`** / opting out of static where documented

Know **why** a route became dynamic when debugging TTFB or cache misses.

## Server Actions

- Colocate **`"use server"`** with a form **`action={...}`** for mutations, or import a **server module** from a Client Component to call an action programmatically.
- **Progressive enhancement** — forms can work before JS loads when using Server Actions with native `<form>` semantics.
- **Validation and auth** — always re-validate on the server; never trust client-only checks for security.

## Route Handlers vs RSC data loading

| | **Route Handler** (`route.ts`) | **Server Component** |
|---|--------------------------------|----------------------|
| **Output** | HTTP response you define | React HTML / RSC payload |
| **Typical use** | Webhooks, REST for non-React clients, streaming custom MIME | Page/layout data for UI |
| **Caching** | `export const dynamic`, `revalidate`, `fetch` in handler | Component `fetch` + segment config |

Use **Route Handlers** when the consumer is not necessarily your React tree.

## Secrets and environment

- **Server-only** env vars (`DATABASE_URL`, private API keys) must **never** be read in Client Components or in modules imported by them on paths that would bundle for the client.
- Prefer **`server-only`** package or clear **module boundaries** so secrets stay in server-only graphs.

## `"use client"` boundary

Move **`"use client"`** as **far down** the tree as reasonable: keep data loading and layout chrome on the server; isolate leaf widgets that need **`useState`**, **`useEffect`**, or browser APIs. Each client boundary is a **separate JS bundle** cost.

## Deployment and runtime

**Server-side** here includes **Node** (default), **Edge** (when configured), and platform adapters (Vercel, Docker, etc.). **Edge** limits Node APIs—document when your data layer is edge-compatible.
